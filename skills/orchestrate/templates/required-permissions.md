# Required permissions for the auto-mode bots

The pipeline bots run in auto mode and SHARE the global allow-list in
`~/.claude/settings.json` (Agent-Teams teammates cannot have a narrower set than
the lead - so read-only is enforced by the CHARTER prompt, not by permissions).

Before launching, diff this list against `permissions.allow` in settings.json and
PRINT any missing entries for the user to approve. NEVER edit settings.json
silently. Changes may require a session restart to take effect.

## Needed allow-list entries (bare-invocation + bash forms as the repo uses)
- `Bash(scripts/safe-push.sh *)` and `Bash(scripts/safe-push.sh)`  (pr-shipper push; bare form, since the repo's `bash scripts/*.sh` rules do not cover it)
- The NON-MERGE `gh pr` subcommands, ENUMERATED (never a blanket `gh pr *` wildcard): `Bash(gh pr view *)`, `Bash(gh pr diff *)`, `Bash(gh pr checks *)`, `Bash(gh pr create *)`, `Bash(gh pr list *)`, `Bash(gh pr status *)`, `Bash(gh pr edit *)`, `Bash(gh pr ready *)`, `Bash(gh pr comment *)`.
- `Bash(gh pr merge *)` - the EXPLICIT MERGE-SCOPED allow-list entry (the sanctioned form per #105, the single entry _missing_allow_entries harvests). This replaces the old "omit merge" posture. The floor deny makes it safe: in a marker-active team session the floor hard-denies the merge CLI regardless of this allow-rule; in a solo/non-marker session (the maintainer's own `/merge-pr`) it lets the merge run prompt-free. (The `gh pr merge:*` and bare `gh pr merge` forms are also doctor-accepted, but only the merge-scoped `gh pr merge *` entry above is harvested - all three forms are NAMED here without a backtick-Perm wrapper precisely so this explanatory prose is not itself harvested as a phantom allow-list entry (the issue #107 doc contract).) A blanket `gh pr *` wildcard is still FORBIDDEN - it grants more than the merge family and hard-fails doctor (see the INVARIANT section below).
- gh-api access is via the P3-F WRAPPERS, NOT a broad gh-api allow-rule (issue #24). `gh-api-get.sh` (read-only; refuses every mutation flag), `gh-codeql-dismiss.sh` (CodeQL dismiss only), `gh-resolve-thread.sh` (resolveReviewThread only), `gh-comment.sh` (PR/issue comment post/reply/inline only; body is data, never the endpoint/method), `gh-codeql-autofix.sh` (CodeQL autofix request only, numeric alert), `gh-delete-branch.sh` (branch ref delete only, validated+url-encoded name) are BUNDLED in the plugin under `scripts/` and invoked as `${CLAUDE_PLUGIN_ROOT}/scripts/<x>.sh`, so they need the PLUGIN-RESOLVED allow coverage described under the scripts entry below (the stable-path `~/.claude/scripts/` rule alone no longer covers the resolved plugin-cache path). REMOVE any broad `gh api` allow-rule from settings.json (the Bash(gh api *) form deliberately written WITHOUT a backtick-Perm wrapper here so the harvester does not re-add it as a phantom entry - issue #107): the broad rule let a bot reach merge-by-API (`gh api -X PUT .../pulls/N/merge`); none of the wrappers can (construction guarantee - each pins its verb + endpoint to one family and accepts no caller-supplied -X/--method, see `../design/DESIGN-p3f-wrapper-inventory.md`). The guard's marker-gated merge-by-API deny stays as a backstop until its separate retirement.
- `Bash(gh issue *)`  (lead files tracking issues)
- `Bash(git *)`  (push is hook-gated against main; safe-push wraps it)
- `Write(//tmp/**)`, `Edit(//tmp/**)`, `Read(//tmp/**)` (+ `//private/tmp/**` on macOS)  (stack file, briefs, triage output)
- the `go` subcommands the gate runs. NOTE: settings typically has only SCOPED entries (`Bash(go build *)`, `go test *`, `go vet *`, `go mod *`, `go tool *`, `go generate *`), NOT `Bash(go *)`. A bare `go run`/`go env`/`go list`/`go clean` in the gate will PROMPT, and an auto bot CANNOT answer -> STALL. Run the gate once in the LEAD session first, capture every `go ...` it issues, and ensure each is covered (or add `Bash(go *)` deliberately). NOTE: `acceptEdits` mode (implementer) does NOT silence Bash prompts.
- the repo's own gate tools, as its `.gates.toml` / gate-runner and pre-push steps invoke them - allow-list whatever the gate actually runs, verified live (a Go repo typically needs `Bash(make *)`, `Bash(golangci-lint run *)`, `Bash(govulncheck *)`; other stacks differ). These are repo-specific, not a fixed set the shared skill mandates.
- GATE-RUNNER (#169): the PR-lifecycle commands + the pre-push hook delegate gate execution to the deployed `gate-runner.py`. It needs the EXPLICIT allow-list entry Bash(python3 ~/.claude/scripts/gate-runner.py *) plus the bare-invocation form Bash(python3 ~/.claude/scripts/gate-runner.py) (both written WITHOUT a backtick-Perm wrapper here, on purpose, so they are NOT auto-harvested into settings.json - the plan mandates the maintainer grant them by hand, exactly like the `gh api`/`gh pr merge` forms above). Why an explicit entry is REQUIRED: the deployed-helper glob `Bash(~/.claude/scripts/*.sh *)` matches only `.sh`, NOT a `.py`, and a `python3 <path>` invocation does not match a path-prefixed glob either - so the runner would otherwise PROMPT (and an auto bot would STALL). The runner is a least-privilege declarative front-end over the repo's own `.gates.toml`/CLAUDE.md gate commands; it neither merges/pushes nor touches the floor, so this grant adds no privilege a contributor lacks. As with every entry here: PRINT it for the user to approve; NEVER silently edit settings.json.
- `Bash(~/.claude/scripts/*.sh *)`  (the DEPLOYED guard at the stable path + any legacy stable-path scripts)
- PLUGIN-RESOLVED HELPERS (#30): the bundled PR-lifecycle helpers (pr-watch.sh, pr-unreplied-comments.sh, pr-read-comments.sh, reply-comment.sh, resolve-threads.sh, cleanup-worktree.sh, patch-coverage.sh, pr-codeql-autofixes.sh, safe-push.sh) and the gh-*.sh wrappers now live INSIDE the plugin under `scripts/` and are invoked as `${CLAUDE_PLUGIN_ROOT}/scripts/<x>.sh`. The harness substitutes `${CLAUDE_PLUGIN_ROOT}` to the install's ABSOLUTE plugin dir, so the literal command the matcher sees is that absolute path - which differs per install AND changes on every plugin update (marketplace cache: `~/.claude/plugins/cache/<marketplace>/<plugin>/<version>/scripts/`; a `--plugin-dir` dev install resolves to the repo's own `scripts/`). NOTE: the maintainer must add ONE allow-rule covering that resolved path, VERIFIED LIVE against this install (mirrors the Slack-MCP verify-before-pin caveat below) - this is documented here, NOT auto-harvested, because the correct glob is install-specific and an over-broad scripts glob must never be wired blindly. A plain `*` in a Bash() spec expands to `.*` (matches across `/`), so a version-agnostic glob like `Bash(bash ~/.claude/plugins/cache/*/scripts/*.sh *)` covers the marketplace cache dir if CC honors `~` (confirm live; else use the absolute `$HOME`-expanded form); a `--plugin-dir` dev run instead needs the repo-path form. CHARTER CAVEAT: `${CLAUDE_PLUGIN_ROOT}` is auto-substituted only in SKILL.md (skill content), NOT in a teammate charter prompt, so the lead resolves it to the concrete scripts dir when spawning a bot - whatever absolute path the lead uses must be the one the allow-rule covers.
- SLACK MCP (maintainer channel, issue #10): when `ORCHESTRATE_SLACK_CHANNEL` is set, the Slack MCP tools must be allow-listed so an auto-mode lead does not STALL on a permission prompt mid-run. The grant token is INSTALL-SPECIFIC: on THIS install the tools are namespaced `mcp__plugin_slack_slack__slack_*` (observed: `mcp__plugin_slack_slack__slack_send_message`, `mcp__plugin_slack_slack__slack_read_channel`), but the plugin-namespace segment VARIES per install - so VERIFY the live tool name before pinning the string (grep `~/.claude/settings.json`, or check the actual MCP tool name in this session) rather than copying the prefix blindly. Grant the consolidated wildcard (`mcp__plugin_slack_slack__slack_*`) - RECOMMENDED, because the maintainer-channel protocol uses MORE than two tools: `slack_send_message` (cards), `slack_read_channel` (inbound since watermark), `slack_add_reaction` (the :eyes: read-receipt), and `slack_get_reactions` (the actionable-emoji answers). The wildcard covers all of them plus any the protocol adds later. If you instead enumerate, grant ALL FOUR - the old `send_message` + `read_channel` pair alone re-prompts on every reaction read/write and defeats the unattended-poll cadence. These tools are DEFERRED-LOAD (they load on demand via ToolSearch), so they may not appear until first use. CAVEAT: `orchestrate-setup.py doctor` is FORMAT-only here - it CANNOT reach MCP tools to confirm the grant, so a missing entry will not be caught by doctor; it surfaces at runtime as the D4 degradation path (send failure -> terminal-only `## ▶ CHANNEL DEGRADED` card). As with every entry in this section: PRINT missing entries for the user to approve; NEVER silently edit settings.json.
- WORKTREE NAVIGATION (no allow-list entry needed): a bot that must operate in an EXISTING worktree uses `git -C <worktree>` + absolute paths (the universal default - no permission, works for any worktree location). `EnterWorktree`/`ExitWorktree` are HARNESS tools, NOT Bash, so they need NO allow-list entry and bypass the auto-mode `cd` prompt; a charter must explicitly authorize their use (the tool requires explicit instruction). VERIFIED (issue #20): from a cwd-pinned teammate, `EnterWorktree path:` only accepts a worktree under the repo's `.claude/worktrees/`, and the path must be in `git worktree list`. NEVER rely on a Bash `cd` into a sibling worktree - it PROMPTS and an auto-mode bot STALLS (it has also been denied outright before).

## Guardrails - the deterministic floor (installed) + the remaining charter-level wall
The merge / push-main / force invariants now HAVE a deterministic floor: `orchestrate-guard.sh` is installed as the PreToolUse `Bash` hook (deny outranks the shared allow-list). See `../design/DESIGN-deterministic-floor.md`. What the floor covers vs what stays charter-level:
- DONE (merge gating, #105 supersedes the 2026-06-06 allow-list-omission): merge is gated in TWO places. (1) `gh pr merge` CLI is NOW a MARKER-GATED FLOOR DENY (`is_pr_merge` in `orchestrate-guard.sh`): a floor deny OUTRANKS the allow-list, so a blanket shadow that accumulates via "always allow" clicks can no longer defeat the gate. The SANCTIONED allow-list entry is the EXPLICIT merge-scoped form: `Bash(gh pr merge *)` (or `:*`/bare) - this lets the maintainer's own `/merge-pr` run prompt-free in a solo/non-marker session, while the floor blocks a bot in a marker-active team session regardless. Do NOT use `Bash(gh pr *)` - it is broader than the merge family and still hard-fails doctor. (2) Merge-by-API is closed by CONSTRUCTION: raw `Bash(gh api *)` is REMOVED from the allow-list (issue #24), and the guard's marker-gated merge-by-API hard-deny stays as a backstop. A GET merge-status read is still allowed via `gh-api-get.sh`. WHY the #105 change: the old "omit merge from the allow-list" posture was defeated by "always allow" clicks re-granting `gh pr *` in settings.local.json (recurring doctor shadow FAIL); moving the gate to the floor fixes it at the root because a deny outranks any allow-rule. `permissionDecision:ask` is still not viable (live-tested: this CC ignores hook ask, honors exit-2 deny only); the floor deny is the only hook-side enforcement mechanism.
- DONE (floor, ALWAYS-ON): `git push`/`safe-push.sh` to `main`/`master`, bare `--force`/`-f` (non-lease), and `--no-verify` are DENIED unconditionally - this closes the safe-push `$@`-forwarding gap (the old hook matched only literal `git push main`).
- STILL CHARTER-LEVEL (deliberately NOT on the floor): mutations the LEAD legitimately needs mid-session (CodeQL dismiss, `resolveReviewThread`, comment/branch-delete) now go through the P3-F WRAPPERS, not raw `gh api -X` (which is removed from the allow-list, so it PROMPTS). Keep the `gh api -X` (raw) + `gh pr review --approve` prohibitions in BOTH read-only charters (pr-triage's is the model - copy it): a read-only bot uses only `gh-api-get.sh`, never a mutating wrapper. Same for `gh pr review/close/edit/ready`, `make remove-worktree/migrate`, and `pkill -f`/`kill` patterns broader than the bot's own PID (they can reap other agents' servers) - charter-forbid + monitor.
- Add explicit "never push main / never --force / never --no-verify / only the head stack entry" lines to the pr-shipper charter (belt-and-suspenders over the floor).
- The Write/Edit secret-file PreToolUse hooks ARE real deterministic guards (keep them).
- Do NOT add `Bash(jq *)` just for stack edits - jq plus a shell redirect can clobber any file; use the Write tool (scoped to /tmp) for stack mutations instead.
- gh-api access is GET-only-by-default by CONSTRUCTION now: raw `Bash(gh api *)` is removed, so reads go through `gh-api-get.sh` (refuses every mutation flag) and each mutation has its own dedicated, endpoint-pinned wrapper. A read-only bot is charter-restricted to `gh-api-get.sh` only.

## INVARIANT: no cascade rule may BROADLY shadow the merge gate (doctor enforces it)
Claude Code UNIONS `permissions.allow` across the WHOLE settings cascade (`~/.claude/settings.json` + `~/.claude/settings.local.json` + project `.claude/settings.json` + project `.claude/settings.local.json`). A broad allow-rule - `Bash(gh pr:*)`, `Bash(gh pr *)`, `Bash(gh:*)`, `Bash(gh *)`, `Bash(*)` - grants more than the merge family and MUST NOT appear in any cascade file. `orchestrate-setup.py doctor` scans every cascade file, simulates each `Bash(...)` allow-rule, and HARD-FAILS (non-zero, naming the file + offending rule) when any BROADER rule would grant merge invocations. A found shadow is a hard failure, not a warning.
- **#105 reconciliation:** an EXPLICIT merge-scoped allow-rule (`Bash(gh pr merge *)`, `Bash(gh pr merge:*)`, `Bash(gh pr merge)`) is ACCEPTED by doctor - it is no longer a shadow. The floor deny backstops it in a marker session (bot still cannot merge); in a solo session it is the intended human-merge path. The doctor diagnostic also prints this note when reporting a true shadow, so users know the sanctioned form.
- BEHAVIORAL RULE (human): if an "allow once" / "always allow" prompt appears for a `gh pr` command, ALWAYS pick "allow once", NEVER "always allow". "Always allow" re-adds a blanket `Bash(gh pr ...)` rule to settings.local.json, which doctor will flag as a shadow on the next `orchestrate up`. The floor deny (#105) means a bot STILL cannot merge even if a shadow sneaks in - but the broader rule is still doctor-flagged because it over-grants every other `gh pr` subcommand.

## DECLINED APPROACHES (rejected grants - do not re-propose)
- RAW `gh api graphql` ALLOW-GRANT - REJECTED (#167) as floor-weakening, the same class as the `gh api -X` closure (issue #24). The GraphQL endpoint accepts arbitrary `mutation { ... }` bodies alongside read queries, and a Bash allow-rule cannot constrain it to read-only - so granting it would hand a read-only bot an arbitrary-mutation surface (the exact failure the wrapper inventory exists to prevent). The read-only `reviewThreads { isResolved }` signal that motivated the ask is ALREADY wrapped inside `pr-unreplied-comments.sh --check-resolved` (the PLUGIN-RESOLVED HELPERS entry above is the correct coverage path). A read-only bot uses `gh-api-get.sh` for GET reads and the endpoint-pinned P3-F wrappers for the specific mutations it is allowed; it never gets raw `gh api graphql`.
